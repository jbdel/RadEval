# Changelog: 2026-04-28 — transformers 5.x upgrade + RadEval 2.1.0

**Branch**: `transformers-v5` → commit onto `hopprai/RadEval:main`.

This changelog documents the transformers 4.x → 5.x migration of RadEval, the bump from 2.0.0 to 2.1.0, and the bundled BERTScore / RadGraph vendors. It lives under `docs/changelog/` which is stripped by `scripts/publish_public.py` and never reaches `jbdel/RadEval`.

## Purpose

`transformers==5.6.2` removes a pile of long-deprecated APIs (`encode_plus`, `batch_encode_plus`, slow tokenizers, `build_inputs_with_special_tokens` on fast tokenizers, `load_in_4bit`/`load_in_8bit`, etc.). Three indirect dependencies of RadEval —

- `bert-score==0.3.13` (last released 2023-02-20, unmaintained)
- `radgraph==0.1.18` (last released 2025-06-18; ships an archived AllenNLP fork)
- `radgraph-xl` checkpoint format (DyGIE `weights.th`, not a native HF model)

— break on transformers v5. Rather than pin `transformers<5` for RadEval, we vendored both `bert-score` and `radgraph` into `RadEval/metrics/*/_vendor/`, applied surgical v5-compatibility patches, and dropped the external deps entirely. All 16 metrics work on transformers 5.6.2 with bit-exact numerical parity against the published paper references (rajpurkarlab/CXR-Report-Metric for RadCliQ-v1; upstream radgraph test fixtures for RadGraph/F1RadGraph).

## Scope

- **Vendored** `bert-score` 0.3.13 at `RadEval/metrics/bertscore/_vendor/` (scorer.py + utils.py + 26 rescale-baseline TSVs, ~13 kB). Dropped the `bert-score` PyPI dependency.
- **Vendored** `radgraph` 0.1.18 at `RadEval/metrics/radgraph/_vendor/` (222 reachable modules, ~2.5 MB including the trimmed AllenNLP + DyGIE subset actually exercised by `RadGraph(...)` and `F1RadGraph(...)`). Dropped the `radgraph` PyPI dependency.
- **Registry**: re-enabled `radgraph`, `radgraph_radcliq`, `radcliq` entries (were commented out during the v2.0.0 intermediate release).
- **Version**: 1.0.1 → 2.0.0 (initial v5-only release, radgraph disabled) → **2.1.0** (radgraph restored via vendoring, bit-exact numerics recovered).

## Patches applied to vendored `bert-score`

In `RadEval/metrics/bertscore/_vendor/utils.py`:

1. `get_tokenizer(model_type)` now passes `add_prefix_space=True` at `AutoTokenizer.from_pretrained` time for RoBERTa/GPT-2/BART-family model types (detected by name-string match in `_needs_prefix_space`). This is the transformers-v5-supported way to get the legacy slow-tokenizer behaviour (the upstream slow RoBERTa tokenizer is removed in v5, so `use_fast=False` is silently ignored; passing `add_prefix_space` at load time is the sanctioned replacement).
2. `sent_encode` simplified — the per-call `add_prefix_space=True` branch is now redundant (it's baked into the tokenizer at load time) and was silently ignored by v5 fast tokenizers anyway. Single `tokenizer.encode(...)` path for every model family.
3. `BERTScorer.__init__` — we dropped the upstream `model2layers` default-lookup table, so callers must pass `num_layers` explicitly (`RadCliQv1/radcliq.py` updated to pass `num_layers=5` for `distilroberta-base`).
4. Removed `build_inputs_with_special_tokens(...)` call in the empty-sentence path (removed from fast tokenizers in v5); replaced with `tokenizer.encode("", add_special_tokens=True)`.
5. Dropped legacy slow-tokenizer class isinstance checks (`GPT2Tokenizer`, `RobertaTokenizer`) — they are no longer importable in v5. Duck-typed on class name instead.

Regression test `tests/test_bertscore.py` (`expected_score = 0.5305 ± 1e-5` on `distilbert-base-uncased`) and `tests/test_radevalbertscore.py` (`0.3462 ± 1e-5` on `IAMJB/RadEvalModernBERT`) both pass bit-identically.

## Patches applied to vendored `radgraph`

Three surgical patches inside the vendored AllenNLP fork:

1. `_vendor/allennlp/data/token_indexers/pretrained_transformer_indexer.py` — upstream called `tokenizer.build_inputs_with_special_tokens(segment)` on every long-sequence split. That method was removed from `TokenizersBackend` (fast tokenizers) in transformers v5. Replaced with cached single-sequence special-token prefix/suffix ids computed once at `__init__` time from `self._allennlp_tokenizer.single_sequence_start_tokens` / `...end_tokens`. Equivalent for single-sequence inputs (DyGIE never calls this indexer in sequence-pair mode).

2. `_vendor/allennlp/data/tokenizers/pretrained_transformer_tokenizer.py` — all five `tokenizer.encode_plus(...)` call sites replaced with `tokenizer(...)` (`__call__`; upstream renamed it in v5). Tokenizers that don't emit `token_type_ids` (ModernBERT) now fall back to zeros so the reverse-engineer loop still works; safe because models with such tokenizers never consume `token_type_ids` downstream.

3. `_vendor/allennlp/common/cached_transformers.py` — strip `add_special_tokens` from `tokenizer_kwargs` before forwarding to `AutoTokenizer.from_pretrained`. In transformers >=4.44 the tokenizer base grew an `add_special_tokens` method whose name collides with the legacy kwarg some AllenNLP configs pass through. AllenNLP owns add-special-tokens at a higher level (`PretrainedTransformerTokenizer`), so dropping it here is safe. Matches the fix in upstream PR #4 (`corentin-ryr/radgraph`), also addresses issue #5.

Cosmetic / plumbing:

- Top-level `radgraph/radgraph.py` renamed to `_vendor/core.py` to avoid self-name collision with the `radgraph` folder under `RadEval.metrics.radgraph`.
- Pinned `_UPSTREAM_RADGRAPH_VERSION = "0.1.18"` as a literal in `_vendor/core.py` (instead of `importlib.metadata.version('radgraph')` which would require a `radgraph` pip distribution we don't install), so the cache directory `~/.cache/radgraph/0.1.18/<model_type>/` is reused by users who already have weights downloaded.
- All internal `radgraph.allennlp.*` / `radgraph.dygie.*` / `radgraph.utils` / `radgraph.rewards` / `radgraph.radgpt` imports rewritten to `RadEval.metrics.radgraph._vendor.*` via a single `sed` pass. 0 residuals.
- Call-site imports: `RadEval/metrics/_radgraph_adapter.py`, `RadEval/metrics/radgraph_radcliq/radgraph_radcliq.py`, `RadEval/metrics/RadCliQv1/radcliq.py` now import from the vendored tree. Dead `RadEval/metrics/RadCliQv1/radcliq_radgraph.py` removed.
