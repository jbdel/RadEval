# Changelog: 2026-04-29 — package rename `RadEval` → `radeval`

**Branch**: `package-rename` → PR into `hopprai/RadEval:main`.

This changelog documents a code-level naming standardization: the
import-name / directory-name / `setup.py` `name=` for the package is
now `radeval` (lowercase), while the human-facing brand "RadEval"
(the tool's name, its README title, paper citations, class
identifier) is preserved. The class `RadEval` and its display name
remain unchanged. It lives under `docs/changelog/` which is stripped
by `scripts/publish_public.py` and never reaches `jbdel/RadEval`.

## Purpose

Python ecosystem convention is lowercase package names. The previous
`pip install RadEval` / `from RadEval import RadEval` shape was an
outlier — unusual to type, inconsistent with every other package in
the same PyPI dependency graph, and slightly confusing because the
module name and the class name were identical. This rename gives us
the canonical pattern: `pip install radeval`, `from radeval import
RadEval`.

## Scope

**Renamed (code-level):**

- **Directory:** `RadEval/` → `radeval/`. Single `git mv`; all 351
  files inside retain full history (100% rename detection).
- **Module file:** `RadEval/RadEval.py` → `radeval/radeval.py`.
- **All Python imports:** `from RadEval` → `from radeval`,
  `from .RadEval` → `from .radeval`, and the one standalone
  `import RadEval` (in `examples/bench_rewards.py`) → `import
  radeval`. The class identifier `RadEval` is preserved everywhere:
  `from radeval import RadEval` is the canonical form.
- **Dotted module paths in strings:**
  `"RadEval.metrics.bertscore.adapter"` in
  `radeval/metrics/_registry.py` tuples → `"radeval.metrics..."`.
  Same for `setup.py::package_data` keys.
- **On-disk path strings** in `ImportError` messages, `scripts/
  update_pricing.py::LLM_PY`, `scripts/publish_public.py` (three
  sites), and vendored-tree comments in `setup.py`.
- **`setup.py`:** `name='RadEval'` → `name='radeval'`.
- **Actionable code blocks** in live docs (README, docs/trl_rewards.md,
  docs/trl_rewards_benchmarks.md, docs/metrics.md,
  docs/hypothesis_testing.md, docs/file_formats.md): `pip install
  RadEval[...]` → `pip install radeval[...]`, `from RadEval import
  ...` → `from radeval import ...`.

**Preserved (human-facing branding):**

- **Class identifiers:** `RadEval` (the class), `RadEvalBERTScorer`,
  `RadEvalBertScoreMetric`, `RadEvalModernBERT`. All carry "RadEval"
  as a brand fragment.
- **HF model IDs:** `IAMJB/RadEvalModernBERT`.
- **GitHub URLs:** `jbdel/RadEval`, `hopprai/RadEval` (the remotes
  themselves have not been renamed; only the Python package).
- **Paper citations:** EMNLP 2025 citation, paper URL, tagline.
- **Prose in README, docs/**, docstrings:** "RadEval is a Python
  framework…", "RadEval metrics as RL rewards", "Base class for all
  RadEval metrics", etc. The tool's name is "RadEval" in prose.
- **Runtime logger name:** `logging.getLogger("RadEval")` in
  `radeval/radeval.py`. Changing it would silently break any user
  filter rules on that logger namespace.
- **Historical changelogs** under `docs/changelog/` — these
  document the state of the repo at the time they were written.
  Rewriting them would falsify the record. New references in this
  changelog and in future changelogs use `radeval/` paths.

## Backwards compatibility

**This is a breaking change for external consumers.** Users with
`pip install RadEval` in their requirements file and `from RadEval
import RadEval` in their code need to update both. The transition is
mechanical:

```diff
- pip install RadEval
+ pip install radeval
```

```diff
- from RadEval import RadEval
+ from radeval import RadEval
```

No temporary alias / shim is provided. The shim option was
considered: a stub `RadEval/__init__.py` in `site-packages` that
does `from radeval import *` with a `DeprecationWarning`. Rejected
because (a) users are the in-tree hopprai/RadEval team + a small set
of external users we can notify directly, (b) maintaining two
import paths indefinitely creates drift risk, (c) the change is
surface-only — no behavior delta — so a one-line mechanical edit is
the full migration.

The public repo (`jbdel/RadEval`) and the GitHub organization remote
(`hopprai/RadEval`) are **not** renamed. Only the Python package
install name and module import path change.

## Publish-script audit

- `scripts/publish_public.py::PRIVATE_DIRS` unchanged. The rename
  flows through to `jbdel/RadEval` as part of the normal publish
  path: private tree has `radeval/`, the public repo will have
  `radeval/`.
- Leak-scan patterns unchanged (they match private-metric symbol
  names, not directory names).
- `strip_private_metric_files` was updated to look for
  `repo_dir / "radeval" / "metrics" / <name>` instead of
  `repo_dir / "RadEval" / "metrics" / <name>` so the strip finds
  the right paths inside the post-rename public clone.
