# RadEval × TRL: Reinforcement-Learning Support Proposal

*A joint report from three investigating perspectives — TRL expert, HOPPR
`vlm_align` expert, and RadEval expert — on how RadEval should integrate with
Hugging Face TRL for GRPO-style radiology-report RL.*

---

## 1. Executive summary

**Recommendation: ship the smallest set of additions that make RadEval a
first-class reward provider for stock `trl.GRPOTrainer`, and otherwise stay out
of the trainer business.**

Concretely, we propose:

1. **Fix and harden `make_reward_fn`** so it matches current TRL conventions
   (absorbs `completion_ids`, `trainer_state`, `log_extra`, `log_metric`,
   `environments` in `**kwargs`; handles conversational
   `completions: list[list[dict]]`; exposes a `key=` argument to disambiguate
   multi-key metrics; caches loaded scorers so two reward functions that share a
   backbone don't load the same 1 GB model twice).
2. **Add a small, optional `CompositeReward`** helper — weighted sum over
   component metrics with renormalization, NaN/Inf guard, and per-metric
   warn-bands — ported from `vlm_align`'s `RadCliQReward`. It stays thin because
   TRL already sums `reward_funcs` natively; `CompositeReward` exists for
   users who want *one* callable + per-component diagnostics at the same scope.
3. **Replace the current "call-shape" demo** with a *real* runnable GRPO
   example against a small Qwen model, plus an optional VLM variant that uses
   stock `trl.GRPOTrainer` with `Qwen2.5-VL` (no custom trainer).
4. **Add `pip install RadEval[rl]`** extras, `tests/test_rewards.py`, and a
   `pytest.importorskip("trl")`-guarded integration smoke test.
5. **Rewrite `docs/trl_rewards.md`** to reflect TRL's current API (GRPO as
   primary, RLOO mentioned, PPO dropped), document `reward_weights`, and link
   to TRL's VLM docs rather than implying RadEval needs to solve VLM training.

Explicitly *out of scope*: a custom GRPO trainer, a VLM adapter abstraction, a
rollout engine, reward-model training utilities, dataset/collator code. All of
these belong in downstream training code (e.g. `vlm_align`) or in TRL itself,
which now covers the VLM case well enough that RadEval does not need its own.

This is a focused, ~500-line PR that turns a vaguely-documented stub into a
production-quality reward surface and ships with a runnable example. That is
the announcement.

## 2. Findings from each expert

### 2.1 TRL expert

**Current TRL GRPO reward API** (v1.3.x):

- TRL calls `reward_fn(prompts=..., completions=..., completion_ids=...,
  trainer_state=..., log_extra=..., log_metric=..., environments=..., **dataset_columns)`
  and expects `list[float]` back (or `None` entries to skip samples).
- **Every dataset column except `prompt`** is forwarded as a kwarg.
- **Conversational datasets** pass `completions` as `list[list[dict]]` (OpenAI
  message format), not strings.
- `async def` reward functions are now first-class (`asyncio.gather`).
- PPO has been relegated to `trl.experimental.ppo`; GRPO is the idiomatic
  choice for 2025/2026. RLOO is a reasonable prompt-only alternative.

**Verdict on RadEval's current `make_reward_fn`:**

- Signature is correct. `**kwargs` absorbs the new TRL-only kwargs without
  blowing up — no change needed there.
- **Missing**: conversational-format handling. If `completions[0]` is a dict,
  we must extract `content`.
- **Missing**: `log_extra` / `log_metric` exposure. Today we collapse to a
  scalar per sample and discard sub-metric breakdowns.

**Composite rewards — TRL already solves this.** TRL natively sums
`reward_funcs` with optional `reward_weights` in `GRPOConfig`, and auto-logs
`reward/{func_name}/mean` and `/std`. So a user can build a weighted RadCliQ by
passing three `make_reward_fn` results with three weights — no custom class
needed.

**VLM — TRL's story is solid now.** Native VLM support in `GRPOTrainer` across
Gemma3, LLaVA-Next, Qwen2-VL/2.5-VL, SmolVLM2. `examples/scripts/grpo_vlm.py`
is a working template. RadEval should *not* ship trainer code or VLM
collators.

**Recommended URLs to link from RadEval docs:**

- <https://huggingface.co/docs/trl/main/en/grpo_trainer>
- <https://huggingface.co/docs/trl/main/en/grpo_trainer#using-a-custom-reward-function>
- <https://huggingface.co/docs/trl/main/en/grpo_trainer#vision-language-model-vlm-training>
- <https://github.com/huggingface/trl/blob/main/examples/scripts/grpo_vlm.py>
- <https://huggingface.co/docs/trl/main/en/dataset_formats>

### 2.2 `vlm_align` / HOPPR RL expert

**Does `vlm_align` use RadEval?** No. A grep of the codebase turns up only
*anti-references* to RadEval: the team deliberately chose the standalone
`radgraph` pip package over "RadEval subprocess" to avoid cold-start per batch,
and an internal design doc states plainly that "`hoppr_vlm/metrics/metrics.py`
is a HuggingFace Trainer eval adapter that returns epoch-level aggregates via a
RadEval subprocess — wrong interface for per-sample GRPO rewards. All three
metrics must be implemented locally."

**That is the central finding:** the team bypassed RadEval precisely because
RadEval's reward surface, until recently, was corpus/epoch-aggregate, not
per-sample online. `make_reward_fn` has since closed most of that gap — our
job now is to finish it.

**Why `vlm_align` has a custom GRPO trainer:**

- The HOPPR VLM takes `images_pixels` / `images_masks`, not `pixel_values`.
- Only the `text_decoder` submodule is LoRA-adapted; the vision encoder and
  QFormer are frozen. This asymmetry requires a `VLMAdapter` abstraction
  (`loaders/base.py`) that TRL's model-class assumptions can't cleanly express.
- A `vision_cache.py` short-circuits the vision encoder across the 16 rollouts
  of the same study — a big speedup that requires a trainer-level seam.
- The model uses `trust_remote_code` with a monkey-patched
  `_init_vision_encoder`.

These are all **training-layer** concerns and none of them belong in RadEval.
Modern TRL (`GRPOTrainer`) would handle a vanilla HF-registered VLM fine.

**Reward-function interface in `vlm_align`:**

```python
Callable[[list[str], list[str]], tuple[list[float], dict[str, list[float]]]]
```

— i.e., returns `(scalars, {"bertscore_f1": [...], "radgraph_f1": [...]})`.
The second element is the key innovation: **per-sample component
diagnostics** survive the reward call, so composites and loggers see component
values separately. TRL's plain `list[float]` discards that.

However, TRL's native `reward_funcs=[f1, f2, f3]` with `reward_weights=[w1,
w2, w3]` gives the same composition *and* auto-logs `reward/f1/mean` etc. So
**we don't need to port the tuple interface** — we just need to make it easy
to register component rewards as separate TRL reward functions. The
`CompositeReward` class we do ship is the "I want one callable" convenience
sugar, not the canonical path.

**What `vlm_align` has that IS worth upstreaming (small, generic pieces):**

| Piece | Where it lives | Port to RadEval? |
|---|---|---|
| Weighted composite w/ renormalization + `allow_partial` | `rewards/radcliq_reward.py` | Yes — thin `CompositeReward`. |
| NaN/Inf hard-raise (names metric + sample index) | `rewards/radcliq_reward.py:138-145` | Yes — utility. |
| Per-metric warn-bands | `rewards/radcliq_reward.py:44-49` | Yes — utility. |
| Group-normalized advantages | `core/advantages.py` (~20 LOC) | **No** — training-layer, belongs in TRL/user code. |

**What to leave behind (HOPPR-specific):** `VLMAdapter`, rollout engine, GRPO
loss, collators, vision cache, the specific `radcliq_v0` weights (a simplified
internal variant of the paper's RadCliQ — avoid re-publishing under a
RadEval-blessed name).

### 2.3 RadEval expert

**What's already in place:** `RadEval/rewards.py` exports `make_reward_fn` (40
LOC, single-metric), with an API-based-metric warning. It's exported from
`RadEval/__init__.py`. Docs exist at `docs/trl_rewards.md`. The example
(`examples/trl_reward_demo.py`) never instantiates TRL — it prints a string
showing what a TRL call *would* look like. Three tests in `tests/test_e2e.py`
exercise `make_reward_fn` against BLEU only; nothing is verified against
actual TRL.

**Bugs / gaps:**

- **Multi-key-metric latent bug.** `make_reward_fn` takes
  `scorer.metric_keys()[0]`. For F1CheXbert that's `f1chexbert_5_micro_f1`,
  which isn't produced in `per_sample=True` mode — the per-sample key is
  `f1chexbert_sample_acc_5`. Silent failure for anyone using
  `make_reward_fn("f1chexbert")`. Need a `key=` argument and/or smarter
  default.
- **No scorer caching.** Two reward functions each instantiate their own
  BERTScore/F1CheXbert/RadGraph scorers — duplicate loads (up to ~1 GB each).
- **`examples/trl_reward_demo.py` is a stub.** It demonstrates nothing a user
  can't learn from the docstring.
- **No `tests/test_rewards.py`.** Only incidental coverage in `test_e2e.py`.
- **No `[rl]` extras** in `setup.py`. Users have to `pip install trl
  datasets` separately and guess compatible versions.
- **Speed table in `docs/trl_rewards.md`** is roughly correct but omits
  `radgraph` and under-warns on `green` (7B local LLM ≈ unusable per-step).

**Where should RadEval stop?** Dataset curation, prompt formatting, chat
templates, trainer configuration, reward-model training, KL/reference-model
handling, distributed launchers — all downstream. RadEval ships the reward
callable, composite helper, scorer caching, and speed/behavior guidance. That
is the whole contribution.

## 3. Recommended scope

**Ship this** (in one PR, aimed at a RadEval 2.2.0 release):

1. **`make_reward_fn` hardening**
   - Handle conversational completion format.
   - Add `key=` arg for multi-key metrics; fix the latent F1CheXbert bug.
   - Add `clip_range=(lo, hi)` and `default_on_error=0.0` (configurable) —
     so a single crashing sample doesn't kill a GRPO step.
   - Optional `log_metric` / `log_extra` forwarding when TRL provides them.
   - Expose `.metric_keys` on the returned callable so TRL logging can name
     the reward.
2. **Scorer caching** — module-level `_SCORER_CACHE` keyed on
   `(metric, frozenset(kwargs.items()))`; cleared via `clear_reward_cache()`.
3. **`CompositeReward` class** — weighted sum with renormalization,
   `allow_partial`, NaN/Inf raise (with metric + sample index), per-metric
   warn-bands; returns a scalar and exposes `.per_metric(...)` for
   introspection. Thin: ≤150 LOC.
4. **`pip install RadEval[rl]`** — adds `trl>=0.x` and `datasets>=2.x`.
5. **Runnable example** — `examples/trl_grpo_quickstart.py`: ~60 lines, small
   LLM (e.g. `Qwen/Qwen2.5-0.5B`), 20-sample synthetic dataset in
   `tests/fixtures/`, `GRPOConfig(max_steps=10)`, dual rewards (`bleu` +
   `bertscore`). Must run on a single modest GPU in < 5 minutes.
6. **Optional VLM example** — `examples/trl_grpo_vlm.py`: uses stock TRL +
   Qwen2.5-VL, no custom trainer. Kept short and clearly marked as
   demonstrating that "RadEval rewards work unchanged for VLM training."
7. **Tests** — `tests/test_rewards.py` covering:
   - Parametrized `make_reward_fn` across all non-API metrics (sanity +
     multi-key coverage).
   - `CompositeReward` weight math, `allow_partial` renormalization,
     NaN raise, warn-band emission.
   - `score_transform`, `clip_range`, `default_on_error` behavior.
   - `UserWarning` for API metrics.
   - Scorer-cache dedup (load count == 1 when two reward fns share a metric).
   - **TRL integration smoke test** behind `pytest.importorskip("trl")`:
     1-step GRPO on a toy model, assert reward fn was called and
     `trainer.state.global_step == 1`.
8. **Rewritten `docs/trl_rewards.md`** — GRPO primary, PPO dropped, RLOO
   mentioned, links to TRL VLM docs, `reward_weights` pattern documented,
   speed table updated and re-ranked.
9. **README updates** — bump the RL section to point to the real example, not
   the stub.

**Do NOT ship** (answering the plan's explicit questions):

- TRL-compatible reward functions — **yes** (already in place, harden it).
- Example GRPO/PPO training scripts — **yes, one LLM + one VLM**, both
  using stock TRL. No PPO.
- Wrappers/adapters around existing metrics — **no new wrappers** beyond
  what `make_reward_fn` already does; the composite helper is the only
  new class.
- VLM-specific trainer utilities — **no**. TRL handles it. Document, don't
  implement.
- Documentation only — **no**; documentation + a single harden-and-fix pass
  is the right shape.

The boundary between RadEval and downstream training code: **RadEval ends at
`reward_fn(completions=..., **kwargs) -> list[float]` (or a composite of
those) and the accompanying metric scorer objects.** Everything that touches
`nn.Module`, `optim`, images, rollouts, KL divergence, or checkpointing lives
downstream.

**LLM-only vs VLM support:** identical reward interface for both — this is
the whole point. The reward function never sees the image. The quickstart
example uses an LLM; the VLM example demonstrates the image-conditioned case
using stock TRL with exactly the same RadEval rewards.

**Should we incorporate `vlm_align`'s custom GRPO trainer?** No. What we port
is the *reward-layer discipline* (NaN/warn-bands, composite with
renormalization), not the trainer. That trainer exists because of HOPPR's
private VLM architecture and is no longer needed for vanilla HF-registered
VLMs now that TRL supports them natively.

## 4. Proposed API / code structure

```python
# RadEval/rewards.py — expanded surface, same import path
from RadEval.rewards import (
    make_reward_fn,          # existing; hardened
    CompositeReward,         # new
    clear_reward_cache,      # new
)
```

```python
# Minimal single-metric reward (TRL GRPO)
from RadEval.rewards import make_reward_fn
from trl import GRPOTrainer, GRPOConfig

reward = make_reward_fn(
    "f1chexbert",
    key="f1chexbert_sample_acc_5",        # NEW: multi-key disambiguation
    clip_range=(0.0, 1.0),                # NEW
    default_on_error=0.0,                 # NEW
    score_transform=lambda x: (x - 0.5) * 2,
)

trainer = GRPOTrainer(
    model="Qwen/Qwen2.5-0.5B",
    reward_funcs=[reward],
    train_dataset=dataset,                # must have "ground_truth"
    args=GRPOConfig(output_dir="out", max_steps=100),
)
trainer.train()
```

```python
# Multi-metric — idiomatic TRL pattern (radeval gives you the callables)
from RadEval.rewards import make_reward_fn

trainer = GRPOTrainer(
    model="Qwen/Qwen2.5-0.5B",
    reward_funcs=[
        make_reward_fn("bertscore"),
        make_reward_fn("radgraph", key="radgraph_partial"),
    ],
    args=GRPOConfig(output_dir="out", reward_weights=[0.6, 0.4]),
    train_dataset=dataset,
)
```

```python
# Composite — one callable, per-component diagnostics preserved
from RadEval.rewards import CompositeReward

radcliq_like = CompositeReward(
    components={
        "bertscore":         0.43,
        "radgraph_partial":  0.32,
        "radeval_bertscore": 0.25,
    },
    reference_column="ground_truth",
    warn_bands={
        "bertscore":         (-0.1, 1.1),
        "radgraph_partial":  (-0.1, 1.1),
        "radeval_bertscore": (-0.1, 1.1),
    },
    allow_partial=False,
)

trainer = GRPOTrainer(
    reward_funcs=[radcliq_like],          # single callable, weighted inside
    ...
)

# Introspect per-component (e.g. for wandb logging hooks):
breakdown = radcliq_like.per_metric(completions=hyps, ground_truth=refs)
# {"bertscore": [...], "radgraph_partial": [...], ...}
```

**`CompositeReward` is small** (~150 LOC). It owes its existence to three
genuine needs TRL's native weighting doesn't cover: (a) renormalization when
components are dropped at runtime, (b) NaN/Inf detection at the reward
boundary, (c) per-metric warn-bands. Users who don't need these can keep
using the plain list-of-rewards pattern.

  available.

### 5.3 Docs walk-through: `CompositeReward` for a RadCliQ-style reward

In `docs/trl_rewards.md`, a short section showing when to prefer a
`CompositeReward` (one logged reward, per-component telemetry via
`per_metric()`) vs. multiple `reward_funcs` with `reward_weights` (automatic
TRL per-function logging). Neither is wrong; users pick the shape.

## 7. Risks / tradeoffs

- **TRL API churn.** TRL has iterated fast; the exact kwargs passed to
  reward functions have changed multiple times in a year. Mitigation: the
  `**kwargs`-absorb pattern is robust; the integration test pins
  `trl>=X.Y.Z` and catches regressions on the CI matrix.
- **Scorer-cache memory.** If a user builds 10 reward functions for 10
  different metrics, that's 10 models loaded into memory forever. Mitigation:
  cache is opt-out via `make_reward_fn(..., cache=False)` and
  `clear_reward_cache()` is a public escape hatch. Document the GB footprint
  of heavy metrics.
- **`CompositeReward` vs TRL's `reward_weights` — two ways to do the same
  thing.** Real tradeoff. Solution: document both clearly, explain when to
  pick which (single-callable diagnostics vs. native per-function logging).
  Keep `CompositeReward` thin so we're not maintaining a parallel
  implementation of TRL's weight math.
- **VLM example requires a real GPU.** It won't run in CI. Mitigation: mark
  it as `slow`, run it manually before releases, have the LLM quickstart as
  the CI-tested canonical example.
- **Bundling `radcliq_v0` would be tempting but wrong.** The HOPPR
  `vlm_align` weights are a deliberately simplified internal variant of the
  published RadCliQ. If users want *paper-RadCliQ*, point them at the
  existing `radcliq` metric in RadEval (which already has its own adapter).
  If they want a composite, they build it themselves — we give them the
  tool, not a branded default.
- **Multi-key metric default selection** (e.g., F1CheXbert) is a
  compatibility concern. Fixing the latent bug may change the numeric
  output of `make_reward_fn("f1chexbert")` from (broken) to (correct).
  Note in changelog as a bugfix, not a breaking change.

## 8. Announcement positioning

**Headline:** *"RadEval 2.2.0: radiology metrics, now as first-class GRPO
rewards."*

**One-paragraph pitch (for blog / tweet / README banner):**

> RadEval 2.2.0 makes every one of its 16+ radiology metrics — BLEU, ROUGE,
> BERTScore, RadGraph, F1CheXbert, RadCliQ, and more — usable as a reward
> function for reinforcement learning with Hugging Face TRL. A single
> `make_reward_fn("radgraph")` call gives you a drop-in `GRPOTrainer`-
> compatible callable; a new `CompositeReward` helper lets you build
> weighted multi-metric rewards with per-component diagnostics and numeric
> safety guards. Works identically for text-only report generation and VLM
> radiology models. Ships with runnable GRPO quickstart examples for both.

**What makes this announcement-worthy without being maintenance debt:**

- A small, well-defined contribution (one module + two examples + docs) —
  not a new "RL framework."
- Closes a *real* gap the HOPPR team hit last year, where they had to
  re-implement BERTScore / SembScore / RadGraph locally because RadEval's
  surface was epoch-aggregate.
- Validates via a runnable integration test on a toy model, so the
  claim "works with TRL GRPO" is verified, not asserted.
- Plays the supporting role: TRL stays the training framework, Transformers
  stays the model framework, RadEval stays the metrics framework. No
  empire-building.
- Opens a natural path to future one-paragraph announcements
  (e.g., "RadCliQ-v1 reward recipe", "radiology RL leaderboard") without
  locking ourselves into maintaining a trainer.

The measure of success: a radiology researcher wanting to GRPO-tune a
report-generation model should be able to `pip install RadEval[rl]`, copy
`examples/trl_grpo_quickstart.py`, swap in their dataset, and be training in
under 15 minutes — with the confidence that what they see in their logs is
exactly the metric the paper published.

default_on_error=0.0 is dangerous.

Also mention/support: RLOO, because it is another online RL method in TRL that can consume reward functions and is simpler/REINFORCE-like. TRL documents RLOO as a supported trainer.
Lightly mention: PPO, but not as a primary example. It exists in the broader TRL ecosystem, but GRPO/RLOO are better positioning for this task. TRL’s current docs frame the library around methods including SFT, GRPO, DPO, reward modeling, etc.
Do not emphasize DPO/ORPO/KTO unless you have preference-pair data. RadEval metrics can help generate preferences, but that is a different workflow than “metric as online reward.”

I’d phrase the plan as:

RadEval exposes TRL-compatible reward functions that work with online policy-optimization trainers such as GRPO and RLOO. The canonical example uses GRPO because radiology-report metrics are naturally computed on sampled completions. The API is trainer-agnostic: any framework that accepts reward_fn(prompts, completions, references, **kwargs) -> list[float] can use the same RadEval rewards.

So: don’t make this “GRPO support.” Make it “TRL-compatible reward support,” with GRPO as the flagship example.
